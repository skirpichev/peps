Abstract
========

This PEP exposes as public the ``__text_signature__`` string attribute and its
format (with minor modifications), being used currently in the CPython to
describe :pep:`362` signatures of builtins.


Motivation
==========

CPython extension modules currently don't have a standartized way to describe
introspection signatures.  The :pep:`7` mention some (informal) specification
of a "signature line", that should be present in the function docstring.  Some
tools like `Sphinx <https://www.sphinx-doc.org/>`__ (see the
`autodoc_docstring_signature
<https://www.sphinx-doc.org/en/master/usage/extensions/autodoc.html#confval-autodoc_docstring_signature>`__
option) are trying to extract signatures from the docstring, "stopping at the
first line that does not look like a signature".  But generally this
information is unavailable for introspection, for example with the
:func:`inspect.signature()`.

The CPython itself uses undocumented ``__text_signature__`` attribute to expose
function signature for the :mod:`inspect` module.  This PEP proposes that
interface as public with few enhancements:

1. Allow type annotations (that can be used also by type checkers).
2. Support for multiple function signatures (see [1]_ for a context).


Specification
=============

The ``__text_signature__`` attribute contains a string value conforming to:

.. rst-backticks error inside of productionlist!
.. productionlist:: builtin-signature
   text_sigs: (`text_sig` NEWLINE)+
   text_sig: "(" [`~python-grammar:parameter_list`] ")" ["->"`~python-grammar:expression`]

which is a subset of syntax, allowed for usual function definitions [2]_ in the
Python.  The :token:`~builtin-signature:text_sig` can't have embedded newlines.
Annotations and default parameter values can only use built-in names [3]_ and
names present in the module scope.

The :class:`type` implements the ``__text_signature__`` as a managed attribute
that takes its value from the docstring (:c:member:`PyMethodDef.ml_doc`) of a
builtin.  The :token:`~builtin-signature:text_sigs` value, if present, must be
in the beginning of the docstring, followed by the four-characters endmarker
``--\n\n``.  Each :token:`~builtin-signature:text_sig` line in the docstring
may be preceed by optional :token:`~python-grammar:funcname` token, used to
keep the function name, for readability.  Both the text signature and the
endmarker are stripped from the managed attribute ``__doc__`` of a function.

Some examples for, respectively, an instance method, a class method, a static
method and a function:

.. code:: pycon

   >>> int.to_bytes.__text_signature__
   (self, /, length=1, byteorder='big', *, signed=False)
   >>> int.from_bytes.__text_signature__
   (cls, /, bytes, byteorder='big', *, signed=False)
   >>> bytes.maketrans.__text_signature__
   '(frm, to, /)'
   >>> math.ldexp.__text_signature__
   '(module, x, i, /)'

The :func:`math.log()` function is an example of a builtin, that may require
multiple signatures which looks like this:

.. code:: pycon

   >>> math.log.__text_signature__
   (module, x, /)
   (module, x, base, /)

The :func:`inspect.signature()` helper will parse the ``__text_signature__``
value, implementing the following algorithm:

* Return :obj:`None` for multiline signature (see next section).
* Decide wheather the given callable is a function, a method or a bound method.
* Then generate an appropriate fake source code:

  * a function definition with empty body for a module function::

      def text_sig: pass

  * a class definition with a suitable decorated method, e.g. for a class method::

      class Fake:
          @classmethod
          def text_sig: pass

* Evaluate fake source code by :func:`exec()` with ``globals`` parameter,
  including the module namespace.
* Call :func:`inspect.signature()` on a generated function/method/bound method and
  return its result.
* :obj:`None` is returned if any from above steps fails.


Open Issues
===========

The PEP doesn't specify how the :mod:`inspect` module should support multiple
function signatures, i.e. wheather there could be a new class
``inspect.MultiSignature``, that may be returned by the
:func:`inspect.signature()` function or this function can be replaced by
``inspect.signatures()`` function that will return a list of
:class:`inspect.Signature` objects.

Extension of the Argument Clinic [4]_ to support new features is also beyond
scope of this PEP and, probably, may be doable via ordinary CPython issues.

This proposal doesn't specify how the function docstring with signature
specification will be kept syntaxically correct an in sync with the actual
description of the function by :c:type:`PyMethodDef` at C level.


Alternatives
============

Using ``__signature__`` attribute
---------------------------------

The :pep:`362` introduced ``__signature__`` attribute: the
:func:`inspect.signature()` returns its content if it isn't :obj:`None`.

We could implement this attribute for functions in extensions.  The getter
function for this attribute will parse the function docstring to get the
:token:`~builtin-signature:text_sigs` value and directly use
``inspect._signature_fromstr()`` helper to built the :class:`inspect.Signature`
object.  This is a minor modification of the current proposal, that doesn't
introduce a new special attribute.

``*.pyi`` stub files
--------------------

Instead we could use MyPy's `stub files
<https://mypy.readthedocs.io/en/stable/stubs.html>`__ to provide introspection
signatures.

The :pep:`561` introduced standartized ways to distribute type information for
extensions with support for ``*.pyi`` stub files.  ``<module name>.pyi`` file
in the same directory as the module could be used to provide :pep:`362`
signatures.  (The stub file for the :mod:`builtins` module could be kept
alongide with the :mod:`inspect` module file.)

Stub files are written in normal Python syntax, leaving out all runtime logic
like variable initializers or function bodies.  Just as for typing, the only
requirement is that interfaces (i.e. function/class declarations) in the stub
file should match interfaces of the extension module.

This solution has a drawback that it mixes support for optional typing hints
(generally, we don't have type annonations in the stdlib) and more basic
support for introspection signatures in extension modules, that already used in
the CPython stdlib e.g. in the :func:`help()` builtin.


Backwards Compatibility
=======================

The current format for the ``__text_signature__`` attribute uses a special
non-Python syntax for the first identifier in the
:token:`~python-grammar:parameter_list` (``$module``, ``$self`` and ``$type``,
respectively, for instance methods, class methods and functions).

This will be deprecated.  Instead, we will use existing introspection
capabilities of the CPython: the ``__self__`` attribute on a callable (which is
missing for instance methods, contains an instance of :class:`type` /
:class:`types.ModuleType` for a class method or a module function, and is
:obj:`None` for a static method).


How to Teach This
=================

The ``__text_signature__`` attribute will be mentioned in the
:func:`inspect.signature` docstring and the "Introspecting callables with the
Signature object" section [5]_ will be expanded also to document its syntax.
The :pep:`7` will list this convention as an alternative for informal
specification of a "signature line".
