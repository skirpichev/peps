Abstract
========

This PEP proposes a new module for integer-related mathematical functions.

Motivation
==========

The :external+py3.14:mod:`math` module documentaton says: "This module provides
access to the mathematical functions defined by the C standard."  Over time the
module was populated with functions that aren't related to the C standard.

Also, the ``math`` module documentaton says: "Except when explicitly noted
otherwise, all return values are floats."  This is no longer true:  *None* of
`comb`, ` factorial`, `gcd`, `isqrt`, `lcm`, `perm`, `floor`, or `ceil` returns
a float, but the docs don’t say so. In a hypothetical `imath`, where they
belong, a similar catch-all: "Except when explicitly noted otherwise, all
return values are ints." could tell the truth once.

The ``imath`` module would parallel the :external+py3.14:mod:`math` module's
other siblings: :external+py3.14:mod:`cmath` and the
:external+py3.14:mod:`statistics`.  It would provide shared context, which
reduces verbosity in the documentaton and conceptual load.  It also aids
discoverability through grouping related functions and IDEs suggesting helpful
completions.


Specification
=============

The PEP proposes copying the follwing integer-related functions to a new
module, called ``imath``:

* ``gcd(a, b)``
* ``lcm(a, b)``
* ``isqrt(n)``
* ``factorial(n)``
* ``comb(n, k)``


Backwards Compatibility
=======================

Aliases in :external+py3.14:mod:`math` would be :term:`soft deprecated` [6]_,
that is, they would remain, but their use would be discouraged.  Thus, there
are no anticipated code breaks.

Open Issues
===========

Module name
-----------

There is already an ``imath`` project on PyPI, but only with two releases
(0.0.1 & 0.0.2), most recently 4 years ago in 2021.

Polling shows ``intmath`` as another popular name.  The argument made was that
the normal mathematical spelling of the imaginary unit is i, which makes imath
ambiguous.  It also has no conflict with any PyPI module.  On the other hand,
perhaps ``intmath`` may be confused with interval math or numerical
integration.

Other proposed names include ``ntheory`` (like SymPy's submodule),
``integermath`` and ``imaths``.

Module scope and possible library members
=========================================

The following is a list of functions that could one day find a home in the
``imath`` library.  New fucntions are not part of the proposal.  They
illustrate the proposed scope of ``imath`` and its potential utility.
Functions proposed in the initial discussion thread include:

* Divisibility & Modular Arithmetic (see also [5]_)

  * ``gcd(a, b)``: Already in ``math``.
  * ``lcm(a, b)``: Already in ``math``.
  * ``powmod(a, b, m)``: Modular exponentiation (already in pow(a, b, m), but
    renamed for clarity).
  * ``is_coprime(a, b)``
  * ``modinv(a, m)``: Modular inverse (or perhaps better yet, ``gcdext``
    below).
  * ``gcdext()`` --- to solve linear Diophantine equation in two variables (the
    :external+py3.14:class:`int` implementation actually includes extended
    Euclidean algorithm).

* Integer Arithmetic

  * ``ceil_div(a, b)``: Ceiling divide, see [2]_, [3]_.
  * ``ilog(n, base)``: Integer logarihm; currently
    :external+py3.14:func:`math.log` has a special handling for integer
    arguments.  It's unique (wrt other module functions) and not documented so
    far. See [4]_.

* Combinatorics and Integer Sequences

  * ``factorial(n)``: Already in ``math``.
  * ``comb(n, k)``: Already in ``math``.
  * ``fibonacci(n)``

* Integer Roots

  * ``isqrt(n)``: Already in ``math``.
  * ``isqrt_rem(n)``: The integer square root and its remainder.
  * ``sqrtmod(n, modulo)``: Square root within a modular group.

* Primality & Factorization

  * ``is_prime(n)``: Optimized for large numbers.
  * ``prime_factors(n)``: Trial division or Pollard’s rho for small inputs.
  * ``primes_between(lower, upper)``: Uses the sieve of Atkin for large ranges
    that start near zero, or an iterative algorithm for small ranges that are
    far from zero.
  * ``totient(n)``: Euler’s totient function.
  * ``next_prime(n)``
  * ``prev_prime(n)``

* Encoding & Number Representations

  * ``gray_encode(n), gray_decode(n)``
  * ``hamming_weight(n)``
  * ``hamming_distance(a, b)``

* Binary Arithmetic

  * ``is_power_of_two(x)``
  * ``next_power_of_two(x)``


Rejected ideas
==============

There was a brief discussion about exposing ``math.isqrt`` as ``imath.sqrt``.
However, ``isqrt`` is ultimately a different function: it is the floor of the
square root.  It would be confusing to give it the same name (under a different
module).


Arguments against
=================

Discoverability
---------------

Beginners looking for a math function may instinctively look for it in the math
module.  Rather than creating a new module, change the documentation for the
math module to read not mention the C standard or floating-point numbers.

Countering this, complex math is still math, but we have a separate module for
that.  Just as with complex numbers, integers should have their own module too.

Limited benefit
---------------

Whereas the ``cmath`` module is justified by its prevention of a lot of slow
type checks, the ``imath`` module has limited benefit in this regard.  The
example given: ``sin`` has a place in ``cmath``, but not ``imath``.
